# 盛兴物业 AI 理解文档

## 1. 这是什么项目

这是一个物业小程序，核心目标不是展示，而是把物业日常业务做成可运营、可追踪、可收口的后台系统。

## 2. 你应该先理解的对象

### 用户侧

- 业主账号
- 认证状态
- 房屋绑定关系
- 缴费账单
- 报修工单
- 投诉建议
- 商城订单
- 客服工单

### 管理侧

- 小区
- 房屋
- 账单
- 催缴日志
- 投诉处理
- 客服处理
- 商城订单
- 通知配置
- 通知日志

## 3. 关键状态

- `USER_STATUS`
  - 0：待审核
  - 1：已通过
  - 8：未通过
  - 9：禁用

- `FEE_STATUS`
  - 0：待缴费
  - 1：已缴费
  - 9：已作废

- `COMPLAINT_STATUS`
  - 0：待受理
  - 1：处理中
  - 2：已回复
  - 9：已关闭

- `TICKET_STATUS`
  - 0：待回复
  - 1：已回复
  - 2：已关闭
  - 9：已归档

## 4. 主要闭环

### 4.1 认证闭环

用户注册 -> 提交认证 -> 管理员审核 -> 开通业务权限

### 4.2 支付闭环

生成票据 -> 前端发起回调 -> 签名校验 -> 更新账单/订单 -> 写入支付日志

### 4.3 账单闭环

创建账单 -> 催缴 -> 缴费 -> 记录催缴和支付日志 -> 可导出

### 4.4 投诉闭环

用户提交 -> 后台分配 -> 后台回复 -> 用户侧可追踪 -> 关闭归档

### 4.5 客服闭环

用户提问 -> FAQ 匹配 -> 工单分配 -> 客服回复 -> 沉淀 FAQ

## 5. AI 修改时的优先级

1. 先保证路由和数据闭环。
2. 再补页面和详情操作。
3. 最后补文档和样式。

## 6. AI 修改时的硬约束

- 不要删除框架公共目录
- 不要把业务逻辑塞进页面层
- 不要把 mock 流程伪装成真实支付
- 不要让通知日志显示成功但实际没有发送
- 不要新增一套和现有 CCMiniCloud 风格冲突的架构

## 7. 常见改动建议

- 新业务先补 `model` 和 `service`
- 管理动作优先放在详情页
- 导出文件统一走云端文件链接
- 通知先记录日志，再接真实渠道

## 8. 你在回答用户时应该怎么说

- 先说当前状态
- 再说缺口
- 再说下一步
- 如果有风险，要明确说明是 mock、占位还是未验签
