# 后台弹窗视觉与文案标准

日期：2026-05-05

## 设计目标

把后台的新增/编辑流程做成“低噪音、高确认度”的操作体验。

## 视觉方向

- 弹窗采用浅色卡片式容器，避免和底层列表抢视觉优先级。
- 标题区更明确，副标题只用一句话说明保存结果或使用范围。
- 底部只保留主要操作和取消操作，减少决策干扰。
- 弹窗宽度按内容自适应，但保持统一的最大宽度和圆角。

## 文案规则

- 标题统一使用“新增 X”或“编辑 X”。
- 底部主按钮统一使用“保存”。
- 次按钮统一使用“取消”。
- 页面入口按钮统一使用“新增 X”。
- 不再使用“新建”“保存修改”“新增配置”等并列写法。
- 若页面内同时存在多个弹窗，footer 文案仍保持同一套“取消 / 保存”，只靠标题区分对象，不在按钮上重复对象名称。

## 产品规则

- 页面首屏只展示入口和列表，不再把完整编辑表单长期挂在列表旁边。
- 保存成功和保存失败都必须反馈。
- 当页面存在两个以上编辑动作时，优先拆成独立弹窗，避免把不同业务表单混在一起。

## 运营检查清单

- 页面首屏是否只保留列表和入口，不再把编辑表单平铺在页面内。
- 弹窗标题是否是“新增 X / 编辑 X”，底部是否统一为“取消 / 保存”。
- 保存成功和失败是否都有提示。
- 如果页面仍保留业务动作面板，是否已经明确它不是独立编辑区。

## 已确认的例外

- `ResidentsPage` 的单条修改仍保留原有轻量交互，因为它是局部属性修改，不是完整表单编辑。
- `OperationsPage` 是详情动作面板，不属于独立新增/编辑区。

## 后续页面处理建议

- 新增编辑入口优先复用 `ModalDialog`。
- 如果表单字段超过一屏，优先拆分模块，再考虑分页或分组折叠，而不是继续加高页面。
