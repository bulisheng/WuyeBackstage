# CloudBase 后台与数据库同步开发规范

> 适用范围：`shengxing-property-core` 及其后续所有后台、数据库、云函数、静态后台站点改动。

## 1. 目标

建立一条固定开发规则，确保任何后台或数据库修改都满足以下要求：

1. 先同步修改 CloudBase 资源，再更新本地代码
2. 本地代码、数据库结构、云函数逻辑保持一致
3. 每次修改完成后立即提交 Git
4. Git 提交记录必须使用中文

这条规范的目标不是增加流程复杂度，而是减少“代码改了、云端没改”“数据库改了、前端没同步”“改完忘记提交”这类不一致问题。

## 2. 适用场景

以下场景必须遵守本规范：

- 修改后台管理页面
- 修改后台管理 API
- 修改云函数中的后台路由
- 修改 MySQL 数据库结构或初始化数据
- 修改 NoSQL 集合、索引、权限规则
- 修改 CloudBase 静态站点后台并重新发布
- 修改任何会影响后台数据展示、权限、菜单、切换器、模块开关的逻辑

以下场景可以按普通开发流程处理，但如果最终影响到后台或数据库，仍要回到本规范：

- 纯前端样式调整
- 文案修改
- 无后端依赖的组件重排
- 仅影响本地草稿的临时实验

## 3. 核心原则

### 3.1 CloudBase 优先

凡是涉及后台和数据库的改动，都必须先确认对应 CloudBase 资源：

- 云函数
- MySQL
- NoSQL 数据库
- 静态网站托管
- 权限配置
- 域名或访问入口

不能只改本地文件而不检查云端状态。

### 3.2 同步修改

本地代码、CloudBase 资源、数据库结构必须同步更新。  
如果某次改动需要同时修改代码和数据库，必须在同一轮开发中完成，不允许只改一半。

### 3.3 每次改动都要提交 Git

每次完成一个可验证的小改动后，必须立即提交 Git。  
不允许长期堆积未提交的后台或数据库改动。

### 3.4 提交信息必须使用中文

所有 Git commit message 必须为中文，且简洁、准确、可读。

示例：

- `新增小区管理表单`
- `完善管理员权限同步`
- `修复后台切换器状态保存`

不允许使用：

- 纯英文提交信息
- 无意义编号，如 `111`
- 无法表达内容的提交信息，如 `update`、`fix`、`change`

## 4. 标准流程

### 4.1 开始修改前

必须先做以下检查：

1. 确认本次改动是否涉及后台、数据库或 CloudBase 资源
2. 确认当前分支和工作区状态
3. 确认需要同步的 CloudBase 目标资源

### 4.2 开发过程中

按以下顺序执行：

1. 修改本地源码
2. 同步 CloudBase 资源
3. 更新数据库结构或数据
4. 更新必要测试
5. 构建或验证
6. 提交 Git

如果本次改动涉及数据库，则必须同步使用 CloudBase 数据库工具完成：

- 结构变更
- 索引变更
- 数据初始化
- 权限变更
- 数据读写验证

### 4.3 完成一个小步后

每完成一个能独立验证的小步，必须：

1. 运行最少一条验证命令
2. 确认结果正确
3. 提交 Git
4. 使用中文提交信息

## 5. CloudBase 必用规则

### 5.1 后台代码同步

只要后台页面、后台 API、云函数路由发生变化，就必须同步检查并更新 CloudBase 相关内容：

- 静态站点托管是否需要重新构建和发布
- 云函数是否需要更新
- 路由是否需要调整
- 权限校验是否需要调整

### 5.2 数据库同步

只要数据库结构、初始化数据、权限表、模块表、角色表发生变化，就必须同步 CloudBase 数据库操作：

- MySQL 结构变更必须同步到 CloudBase MySQL
- NoSQL 结构变更必须同步到 CloudBase NoSQL
- 初始化数据必须同步写入云端
- 变更后必须验证读写结果

### 5.3 不允许只改本地

以下行为禁止：

- 只改本地 SQL 文件，不同步云端
- 只改本地后台页面，不重新构建静态站点
- 只改本地云函数，不检查部署状态
- 只改数据模型，不同步数据库结构

## 6. Git 提交流程

### 6.1 提交频率

要求是“每次修改一次就提交一次”。  
这里的“修改一次”指一个完整、可验证、范围清晰的小步。

### 6.2 提交前检查

提交前必须确认：

- 本次改动已经验证
- 没有把无关文件一起提交
- 没有把构建垃圾文件误提交
- 提交信息是中文

### 6.3 提交信息规范

推荐格式：

- `新增...`
- `完善...`
- `修复...`
- `同步...`
- `重构...`

建议尽量描述业务结果，不要只描述文件名。

## 7. 验收标准

只要满足以下条件，就认为该规范执行到位：

1. 后台改动都能在 CloudBase 上找到对应同步结果
2. 数据库变更都有对应的云端结构或数据更新
3. 每次改动后都有单独 Git 提交
4. 所有提交信息都是中文
5. 不再出现“本地改了但云端没同步”的情况

## 8. 例外情况

以下情况可以临时不提交，但结束后必须补提交：

- 正在排查问题的中间态
- 正在准备一组连贯的小步改动
- 正在修改会被立即回滚的实验代码

例外不能成为常态。  
任何进入可验证状态的后台/数据库改动，都必须立刻补齐 CloudBase 同步和 Git 提交。

## 9. 实施建议

建议把这条规范作为后续所有后台和数据库工作的默认流程：

1. 先确认 CloudBase 资源
2. 再改代码和数据库
3. 验证通过后立刻提交
4. 提交信息写中文

如果后续需要把这条规范进一步程序化，可以再补充：

- 统一的提交信息模板
- 后台/数据库变更清单
- CloudBase 同步检查清单
- 提交前自动校验脚本
