# 小区模块开关与动态首页设计说明

> 目标：补齐第一阶段最后两块底座能力。

## 背景

当前工程已经完成了：
- `communities` 小区体系
- 后台小区切换器
- 管理员与权限体系
- `admin_community_permissions` 小区权限表
- `module + action` 的后端权限校验
- 操作审计日志

但第一阶段仍缺两块关键能力：
- 小区模块开关
- 小程序首页动态入口

本设计只覆盖这两个缺口，不扩展到无关的新业务。

## 目标

### 必须达到的结果
- 每个小区都能配置自己启用的模块。
- 新建小区时，模块默认全部开启。
- 后台可以在“权限管理”页中通过一个独立 Tab 维护模块开关。
- 后端接口可以真正拦住未启用模块的访问。
- 小程序首页可以按当前小区模块和用户状态动态展示入口。

### 不做的事情
- 不重做现有管理员/角色体系。
- 不把模块开关拆成单独主菜单页。
- 不在这一轮里做按钮级更细的前端权限矩阵重构。

## 设计方案

### 1. 数据模型：`community_modules`

采用“每小区一条模块记录”的方案。

建议字段：
- `id`
- `community_id`
- `module_key`
- `module_name`
- `enabled`
- `sort`
- `created_at`
- `updated_at`

建议约束：
- `UNIQUE(community_id, module_key)`
- `INDEX(community_id, enabled)`

这个结构的好处是：
- 后台矩阵页好展示
- 审计和排查简单
- 后端校验可以直接按小区 + 模块查询
- 后续扩展到按钮级或入口级也不需要改存储模型

### 2. 模块目录

模块目录由代码维护，数据库只存启停状态。

建议首批模块键：
- `dashboard`
- `owners`
- `communities`
- `permissions`
- `repairs`
- `fees`
- `complaints`
- `notices`
- `announcements`
- `audit`

后续如果新增业务模块，只需要补目录和初始化逻辑。

### 3. 默认策略

新建小区时：
- 自动补齐全部模块记录
- 全部 `enabled = 1`

后台可以随时关闭某个模块，但默认状态必须是“全开”。

这样做的原因：
- 新小区初始化成本最低
- 不会出现“新建小区后功能缺失”的空白态
- 后台接手后再按运营需要关闭即可

### 4. 后台模块开关页

模块开关页放在现有“权限管理”页中，作为一个 Tab。

建议结构：
- Tab 1：管理员
- Tab 2：小区权限
- Tab 3：模块开关
- Tab 4：操作审计

模块开关页内容建议：
- 顶部：当前小区选择器
- 中部：模块开关列表或矩阵
- 底部：批量启用 / 批量关闭 / 全量重置

交互规则：
- 切换小区后立即刷新该小区的模块状态
- 单个模块可切换启用或关闭
- 批量操作需二次确认
- 新建小区默认全开，后台只负责关闭

### 5. 后端校验

需要补一层统一校验：`checkModuleEnabled`

校验顺序建议保持：
1. 先确认当前小区存在且可访问
2. 再确认目标模块是否启用
3. 最后做角色和动作权限判断

这样可以保证：
- 模块未启用时，接口直接被拦住
- 不是只有前端隐藏入口，后端也会拒绝请求
- 现有 `module + action` 权限体系继续保留

### 6. 小程序首页动态化

首页不再写死入口，而是根据“当前小区 + 模块开关 + 用户状态”生成。

首页渲染顺序建议：
1. 读取当前小区
2. 获取该小区启用的模块
3. 结合用户状态决定入口展示
4. 输出首页模块列表

展示规则：
- 模块关闭则完全不显示
- 模块开启但用户状态不满足时，可显示灰态或引导态
- 对用户而言，一个首页适配所有小区

## 数据流

### 模块开关后台
1. 管理员选择小区
2. 前端请求模块列表
3. 后端返回该小区全部模块及状态
4. 管理员修改状态
5. 后端落库并记录审计

### 小程序首页
1. 用户进入首页
2. 系统识别当前小区
3. 拉取该小区模块开关
4. 根据用户状态生成入口
5. 渲染动态首页

## 接口建议

### 后端接口
- `admin/community/module/list`
- `admin/community/module/save`
- `admin/community/module/batch_save`
- `admin/community/module/reset`
- `community/module/list` 或首页使用的公开/半公开查询接口

### 校验函数
- `checkCommunityPermission`
- `checkModuleEnabled`
- `checkRolePermission`

### 前端请求
- 权限管理页新增模块开关 Tab 的数据加载与提交
- 小程序首页新增动态模块加载逻辑

## 初始化和迁移

新建小区时必须补齐模块记录。

建议初始化逻辑：
- 以代码里的模块目录为准
- 对该小区逐条 upsert
- 默认全部启用

如果后续新增模块：
- 不要手工只改后台页
- 要同时补初始化逻辑和历史小区补齐逻辑

## 测试范围

### 后端测试
- `community_modules` 初始化是否补全
- 单个模块启用 / 关闭是否生效
- 未启用模块是否能拦住接口调用
- 角色权限和模块权限叠加后是否符合预期

### 前端测试
- 权限管理页 Tab 切换是否正常
- 模块开关是否能按小区刷新
- 首页入口是否会随模块状态变化

### 回归重点
- 不能影响当前已经完成的权限管理、审计和小区切换器
- 不能让新模块把老模块显示逻辑打乱

## 风险与注意事项

- 模块目录变更时必须同步初始化逻辑，否则历史小区会缺记录。
- 不能只做前端隐藏，必须同时做后端拦截。
- 新建小区默认全开，避免运营人员在创建后看不到基础功能。
- 模块开关页放在“权限管理”Tab 里，避免后台菜单继续膨胀。
- 小程序首页动态化时要保持入口布局稳定，避免不同小区视觉差异过大。

## 实施顺序

1. 新增 `community_modules` 表和初始化逻辑
2. 在“权限管理”页新增模块开关 Tab
3. 在后端统一入口补 `checkModuleEnabled`
4. 把首页改成动态模块入口版
5. 补测试和审计

## 第一阶段完成标志

当以下条件都满足时，第一阶段才算真正完成：
- 后台可切换小区
- 不同小区数据隔离
- 不同角色可见范围不同
- 不同小区功能可关闭或开启
- 后端会拦截未启用模块
- 小程序首页能按小区动态展示入口
