# 盛兴物业框架文档

## 1. 总体结构

项目采用 CCMiniCloud 风格的分层结构：

- `model`：数据集合和字段定义
- `service`：业务逻辑
- `controller`：云函数入口层
- `route.js`：路由注册表
- `miniprogram/projects/house/pages`：小程序页面

## 2. 核心目录

```text
cloudfunctions/mcloud/project/house/
├─ model/
├─ service/
├─ controller/
└─ public/
```

```text
miniprogram/projects/house/pages/
├─ default/
├─ community/
├─ fee/
├─ task/
├─ complaint/
├─ service/
├─ mall/
├─ customer/
├─ vote/
├─ activity/
├─ my/
└─ admin/
```

## 3. 请求链路

1. 小程序通过 `helper/cloud_helper.js` 调用云函数。
2. 云函数入口 `mcloud` 读取 `route`。
3. `route.js` 将路由映射到控制器方法。
4. 控制器做参数校验和权限判断。
5. 服务层执行业务并写入模型。

## 4. 数据模型

项目的数据集合按业务边界拆分：

- `house_fee_bill`
- `house_fee_pay_log`
- `house_fee_reminder_log`
- `house_complaint`
- `house_customer_ticket`
- `house_mall_goods`
- `house_mall_order`
- `house_notice_config`
- `house_notice_log`
- `house_user_house`

## 5. 认证与权限

- 未认证用户只能访问基础内容。
- 认证通过后才能进入缴费、报修、投诉、物业服务、商城和客服。
- 绑定房屋后才允许进入需要房屋上下文的业务。

## 6. 支付与通知

### 6.1 支付

- 先由服务端生成支付票据。
- 前端提交回调时必须携带签名字段。
- 后端校验签名通过后才落库。

### 6.2 通知

- 通知模板定义在 `public/notice_templates.js`
- 通知配置由后台管理页维护
- 通知日志写入 `house_notice_log`

## 7. 后台管理约定

- 列表页只负责筛选和跳转
- 详情页负责状态流转和业务操作
- 导出统一走云端文件
- 催缴、回复、分配这类动作必须记录日志

## 8. 新增模块的接入顺序

1. 先加 `model`
2. 再加 `service`
3. 再加 `controller`
4. 再注册 `route`
5. 最后补页面和文档

## 9. 修改边界

- 不要随意改现有页面路径
- 不要破坏 `framework/` 的公共封装
- 不要把业务逻辑堆到小程序页面里
- 不要跳过路由层直接访问数据库
