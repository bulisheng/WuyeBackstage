## 1. 项目一句话

这是一个 **多小区 SaaS 物业系统**，不是单纯的小程序，也不是普通后台。

核心目标是：

- 一套系统支持多个小区
- 不同小区数据隔离
- 不同小区功能不同
- 不同角色看到不同范围
- 后台能管、前端能变、接口能拦

## 7. 现在的权限判断顺序

后端现在应该始终按这个顺序理解：

1. 先判断当前小区是否可访问
2. 再判断该小区模块是否启用
3. 再判断角色是否允许这个动作
4. 最后才执行具体业务

不要把权限判断只放前端。

## 8. 当前工程的硬约束

这是接手时最容易踩坑的地方：

- 不要只改本地，不同步 CloudBase
- 不要改完后台和数据库却不提交 Git
- 不要把模块开关只做成隐藏菜单
- 不要把页面写死成单小区
- 不要让某个接口绕过小区和模块校验
- 不要把角色默认菜单和小区覆盖逻辑搞反

## 9. 新对话里应该怎么判断任务优先级

如果用户问“先做什么”，建议按这个顺序：

1. 先保证路由和数据闭环
2. 再补页面和详情操作
3. 再补样式和交互
4. 最后补文档和整理

## 11. 简化版总结

- 第一阶段已经完成
- 第二阶段从缴费、报修、统一工单开始
- 后续所有改动都要继续遵守多小区 SaaS、权限、模块开关和审计原则

## 12. 第二阶段提醒

- 现在不要把缴费做成单纯列表页，必须围绕账单、支付、支付记录、催缴这条链路来做。
- 现在不要把报修做成孤立页面，后面要能落到统一工单底座。
- 新增后端路由时，记得同步 `permission_engine.js`，否则模块拦截会漏。
- 新增数据库表时，记得同步 CloudBase MySQL 和 `database/mysql/schema.sql`。
- 新增完成后，记得同步更新这个 AI 理解文档，不要让下一位 AI 读到旧状态。
- 后台现在靠登录态决定可见界面，超级管理员负责创建和管理其他管理员，不要再依赖“选择操作对象”这个旧入口。
