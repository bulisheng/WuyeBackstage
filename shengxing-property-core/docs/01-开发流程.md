# 社区物业 SaaS 系统 - 开发流程最终版

## 1. 项目定位

本项目不是单一小程序，而是一个多小区物业 SaaS 系统。

系统核心能力包括：

- 小程序端服务入口
- 后台管理端
- 多小区数据隔离
- 小区功能模块开关
- 业主认证
- 缴费
- 报修
- 投诉建议
- 物业服务
- 商城
- 在线客服
- 投票
- 活动
- 通知系统
- 数据分析

当前已完成：

- 业主登录
- 房屋绑定
- 业主认证

后续开发必须围绕“多小区 SaaS 化运营”设计。

---

## 2. 核心设计原则

### 2.1 所有业务数据必须属于某个小区

所有核心业务表必须包含：

```sql
community_id
```

包括但不限于：

- houses
- user_houses
- bills
- bill_items
- payments
- tasks
- task_logs
- task_images
- products
- orders
- order_items
- activities
- activity_registrations
- votes
- vote_records
- notifications
- banners
- notices

任何业务数据不得脱离 `community_id` 存在。

---

### 2.2 后台必须支持切换小区

后台管理员登录后，系统返回该管理员可管理的小区列表。

后台顶部必须有“小区切换器”。

切换小区后：

- 当前后台全局保存 `current_community_id`
- 所有页面只展示当前小区数据
- 所有新增数据自动归属当前小区
- 所有编辑、删除、查看操作必须校验数据是否属于当前小区
- 不能通过修改接口参数访问其他小区数据

---

### 2.3 后端权限不能依赖前端

前端隐藏菜单只是体验优化。

真正权限必须由后端控制。

每个接口都必须校验：

1. 当前管理员是否登录
2. 当前管理员是否拥有 `current_community_id` 权限
3. 当前管理员角色是否允许访问当前模块
4. 当前小区是否开启该模块
5. 当前数据是否属于 `current_community_id`

---

## 3. 小区功能模块开关

### 3.1 设计目标

每个小区可以独立配置开启哪些功能。

例如 A 小区开启：

- 缴费
- 报修
- 投诉
- 在线客服

B 小区开启：

- 报修
- 商城
- 活动

两个小区的小程序入口、后台菜单、接口权限都可以不同。

---

### 3.2 模块 key 设计

```txt
payment              生活缴费
repair               报事报修
complaint            投诉建议
property_service     物业服务
mall                 盛兴严选 / 商城
customer_service     在线客服
vote                 投票
activity             社区活动
notice               公告
banner               首页 Banner
```

---

### 3.3 community_modules 表

```sql
community_modules

id
community_id
module_key
module_name
enabled
sort_order
config_json
created_at
updated_at
```

字段说明：

- `module_key`：系统固定 key
- `module_name`：当前小区自定义显示名称
- `enabled`：是否启用
- `sort_order`：首页或菜单展示排序
- `config_json`：模块个性化配置，例如客服配置、支付配置、活动限制等

---

### 3.4 模块关闭规则

当模块关闭时：

小程序端：

- 首页隐藏入口
- 页面直接访问时提示：`当前小区暂未开通该服务`
- 不允许新增数据
- 历史记录允许查看

后台端：

- 后台隐藏该模块菜单
- 历史数据可以查看
- 不允许新增业务数据
- 管理员仍可在系统配置中重新开启

接口端：

- 新增、提交、支付、报名、投票等写操作必须拒绝
- 查询历史数据可以允许，但必须校验权限

## 4. 角色与小区权限

### 4.1 后台角色

```txt
super_admin       超级管理员
community_admin   小区管理员
customer_service  客服
repair_staff      维修人员
finance           财务
operator          运营
```

---

### 4.2 角色权限说明

#### 超级管理员

- 可查看所有小区
- 可切换所有小区
- 可管理模块开关
- 可管理后台账号
- 可查看平台级数据

#### 小区管理员

- 只能查看授权小区
- 可管理授权小区内全部业务
- 可查看该小区数据看板
- 可配置该小区基础信息

#### 客服

- 只能查看授权小区
- 只能处理投诉、客服、部分报修工单
- 不能查看财务敏感数据
- 不能修改小区配置

#### 维修人员

- 只能查看授权小区
- 只能查看分配给自己的报修工单
- 可更新处理状态
- 可上传处理记录
- 不能查看财务和商城数据

#### 财务

- 只能查看授权小区
- 只能管理账单、缴费、支付记录、催缴
- 不能处理工单分配
- 不能配置商城商品

#### 运营

- 只能查看授权小区
- 只能管理商城、活动、投票、公告、Banner
- 不能查看财务敏感数据

---

### 4.3 admin_community_permissions 表

```sql

id
admin_id
community_id

```

说明：

- 一个管理员可以绑定多个小区
- 一个管理员在不同小区可以拥有不同角色

示例：

```txt

B 小区：维修人员

C 小区：小区管理员
```

## 8. 首页动态规则

首页不是固定页面，必须根据以下条件动态渲染：

1. 当前小区开启的模块
2. 当前用户认证状态
3. 当前用户是否有待缴账单
4. 当前用户是否有进行中工单
5. 当前小区是否有公告、活动、Banner

## 10. API 开发强制要求

每个后台接口必须接收或识别：

```txt
current_community_id
```

后端必须通过登录态判断管理员是否拥有该小区权限。

禁止信任前端传入的 `community_id`。

所有查询必须自动追加：

```sql
WHERE community_id = current_community_id
```

所有新增必须自动写入：

```sql
community_id = current_community_id
```

---

## 11. 模块权限中间件

建议实现统一中间件：

```txt
checkCommunityPermission
checkModuleEnabled
checkRolePermission
```

每个接口进入业务逻辑前必须经过：

1. 登录校验
2. 小区权限校验
3. 模块开关校验
4. 角色权限校验

---

## 12. 最终验收标准

一个模块只有满足以下条件，才算完成：

- 小程序端有入口
- 后台端有管理页面
- API 有权限校验
- 数据有 `community_id`
- 支持模块开关控制
- 支持状态流转
- 支持通知
- 支持日志
- 支持空状态、加载状态、错误状态
